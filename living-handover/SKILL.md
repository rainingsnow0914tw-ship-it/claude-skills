# living-handover

## 核心概念

大多數 handover 的問題，不是「沒有交接手冊」，
而是**手冊只在最後一刻才被生成**。

這會導致：
- 前面做過的關鍵決策已經散失在聊天記錄裡
- AI 到了要整理時，自己也忘了脈絡
- 最後輸出的手冊只有大綱，缺少真正可接手的細節
- 新對話框還是得重新解釋一次

所以這個技能的基本立場是：

> **handover 不是收尾動作，而是整段工作流程中的持續維護行為。**

也就是說，Claude 不應只在「快換框了」才輸出交接，
而應該在整個任務進行過程中，逐步累積一份可更新的活手冊。

---

## 這份手冊應該記什麼

`living-handover` 的目標不是記錄所有聊天廢話，
而是只保留**未來真的會影響接手與延續的資訊**。

應優先記錄的內容包括：

### 1. 當前目標
- 目前這輪在做什麼
- 任務的短期目標與中期目標
- 成功判準是什麼

### 2. 已完成事項
- 已經做完哪些步驟
- 哪些檔案 / prompt / 方案已經產生
- 哪些決策已鎖定

### 3. 關鍵決策與原因
- 為什麼選 A 不選 B
- 哪些限制來自平台 / 使用者偏好 / 舊系統相容
- 哪些路徑被證明不可行

### 4. 當前狀態
- 現在卡在哪裡
- 哪些東西已測過
- 哪些假設還沒驗證

### 5. 下一步
- 下一個 Claude / 協作者應該先做什麼
- 有沒有待確認事項
- 哪些風險要先注意

---

## 什麼時候要更新手冊

Claude 應在以下情況主動更新，而不是等使用者提醒：

### 1. 任務進入新階段
例如：
- 從發想進入實作
- 從實作進入 debug
- 從 debug 進入部署
- 從探索進入定稿

### 2. 做出重要決策
例如：
- 選定技術路線
- 放棄某個方案
- 確認某個 prompt / 架構 / 命名
- 確認某個 bug 的真正原因

### 3. 產出可交付物
例如：
- 新檔案
- 新版 prompt
- 新版 PRD
- 新版圖表 / 介面 / workflow

### 4. 對話已經很長
當 Claude 判斷上下文已經累積太多，
若再不整理，下一段就容易失真或遺漏，
應主動生成一次 handover patch。

---

## 更新形式

這個技能不要求每次都重寫整份手冊。
Claude 應依情境選擇：

### A. Patch 更新
當只是補幾個新進展時，直接補充：
- 新完成內容
- 新決策
- 新待辦

### B. 完整版重整
當使用者說「生成交接手冊」「準備換框」時，
再輸出一份結構完整、可直接貼去新對話框的 handover 文檔。

---

## 推薦手冊結構

建議使用以下結構：

### 1. Project / Task 名稱
目前在處理哪個專案或哪個子任務。

### 2. Goal
這輪要達成什麼。

### 3. Completed
已完成事項。

### 4. Decisions
已確定的關鍵決策與原因。

### 5. Current Status
目前進度、卡點、已驗證 / 未驗證內容。

### 6. Next Actions
接下來應做的步驟。

### 7. Important Context
不能丟的上下文，例如：
- 命名規則
- 檔案位置
- 使用者偏好
- 技術限制

---

## Claude 的寫法原則

### 要做到
- 精準，只記真正影響延續的資訊
- 結構化，讓新接手者能快速掃描
- 可更新，不要每次都寫成僵硬報告
- 可貼用，必要時能直接丟進新對話框

### 不要做
- 不要把整段聊天逐字摘要
- 不要堆一堆空泛句子
- 不要只寫「我們剛才做了很多事」這種無效總結
- 不要遺漏「為什麼這樣決定」

因為真正最容易在換框時消失的，
往往不是做了什麼，而是**為什麼這樣做**。

---

## 和其他技能的分工

### 它負責
- 維護整個任務 / 專案層級的可接手脈絡
- 為新對話框或新協作者提供延續入口
- 保存進度、決策、卡點、下一步

### 它不負責
- 把局部設計知識寫進程式碼本體
- 為函數補 docstring
- 記錄某個單一函數的細節限制

那些屬於 `code-documentation`。

---

## 成功判準

這個技能做得好時，應達成：

- 就算對話中斷，也能快速恢復工作
- 新 Claude 不需要從零開始理解
- 使用者知道目前做到哪、下一步是什麼
- 決策理由沒有隨著上下文被沖掉

---

## 一句話提醒 Claude

> 不要等到快失憶才寫交接；真正好的 handover，是在整段工作過程中一直活著。
