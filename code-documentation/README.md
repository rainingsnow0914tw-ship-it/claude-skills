# 🐱 code-documentation skill

**自文件化代碼規範** — 讓 Claude 自動把設計知識鎖進代碼注釋，防止換對話框後功能退化。

---

## 這個技能解決什麼問題？

當 AI 幫你寫程式時，最麻煩的不是「寫出功能」，而是：

- 下一次開新對話框，AI 不知道當初為什麼這樣寫
- 函數內部的設計意圖只存在聊天記錄裡
- 換模型 / 換人接手時，只看到 code，不知道 rule
- debug 時容易把原本重要的 guardrail 洗掉

這個 skill 的目的，就是把**設計知識一邊做一邊寫進代碼本體**，讓 code 不只是能跑，還能自己說明：

- 這個函數在幹嘛
- 為什麼這樣做
- 哪些地方不能亂改
- 它和前後流程怎麼接

也就是讓 code 變成一種 **self-documenting working memory**。

---

## Claude 應該做什麼？

啟用這個技能後，Claude 在寫或改 Python 程式時，應自動：

1. 為每個新函數補上清楚註釋
2. 如果是重要函數，補上「設計意圖」與「不可亂改點」
3. 如果是 debug 修補，註明：
   - bug 原因
   - 修補原理
   - 為何不用別的方法
4. 對關鍵資料流補充 inline comment
5. 保證註釋不是廢話，而是未來交接真的有用的資訊

---

## 適用情境

這個 skill 特別適合：

- Python backend
- agent workflow
- API bridge
- 資料處理 pipeline
- 多 AI 共創後容易斷上下文的專案

---

## 成功標準

使用完這個 skill 後，理想狀態是：

- 只看 code 也能大致看懂系統意圖
- 就算換新對話框，Claude 也較不容易亂改壞
- 你未來回頭看，不會覺得這段像天書
- 司機先生如果只看註釋，也能抓到大方向

---

## 搭配建議

它很適合和：

- `living-handover` 一起用 → 一個把知識寫進 code，一個把全局狀態寫進手冊
- PRD / architecture doc 一起用 → 外部規格 + 內部註釋雙保險
